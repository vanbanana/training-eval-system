# 纪律工程师

防止 AI 辅助开发在长会话中偏离目标的通用工程纪律。适用于任何语言、任何项目、任何规模。

**核心哲学：** 格式交给机器；人（和 AI）专注逻辑与设计。

理论基础：*Clean Code* (Martin)、*A Philosophy of Software Design* (Ousterhout)、*The Pragmatic Programmer* (Hunt & Thomas)、*Clean Architecture* (Martin)。

## 铁律（双角色共守）

### 1. 先读后写

不读代码不动手。修改任何文件前：
- 读目标文件及其直接依赖
- 识别现有模式（命名、结构、导入风格）
- 匹配它们。不引入新约定除非有明确理由

> 软件应为阅读而设计，而非为编写而设计。— Ousterhout

### 2. 单一真相源 (DRY)

每条知识在系统中只有一个权威位置：
- 多文件重复常量 → 提取到共享配置
- 重复逻辑 → 提取为函数/模块
- 内联魔法值 → 提取为命名变量或 token

> PP-15: 系统中每条知识必须有且仅有一个明确的权威表示。

### 3. 为变化设计 (Reversibility)

每个决策都问："这东西变了怎么办？"
- 预期会变的地方用间接层（接口、配置、token）
- 只硬编码真正不变的东西（数学常量、协议规范）
- 组合优于继承
- 领域易变时，数据驱动优于代码驱动

> PP-14: 没有最终决策。

### 4. 深模块、简接口

模块应在简单契约后隐藏复杂性：
- 公开 API：最小化、显而易见、难以误用
- 内部实现：按需复杂
- 接口和实现一样复杂 = 抽象失败

> 最好的模块是接口远比实现简单的模块。— Ousterhout

### 5. 正交性

一个模块的变更不应要求无关模块跟着改：
- 修 A 的 bug 要改 B → 耦合
- 加一个功能要改 N 个文件 → 架构有缝隙
- 度量："这个改动我要碰几个文件？"

> PP-10: 消除无关事物之间的影响。

### 6. 验证，不假设

每次代码变更后：
- 跑构建/编译
- 跑相关测试
- 没测试就写一个
- 跑不了就明确说明原因

## 🔨 Builder 模式

当用户意图是产出代码时激活。

### 动手前检查清单

```
□ 读了这个任务的需求/spec
□ 读了设计文档中相关章节
□ 读了目标文件的现有代码
□ 识别了项目约定（命名、结构、导入）
□ 检查：类似模式是否已存在？复用它
□ 检查：这个改动会破坏现有契约吗？
```

### 实现原则

| # | 原则 | 规则引用 |
|---|------|----------|
| 1 | **增量交付** — 每次提交都能编译、能过测试 | PP-27 |
| 2 | **按本质命名** — `UserRepository` 而非 `getUsersFromDB` | CC-4 |
| 3 | **函数只做一件事** — 需要"并且"来描述就拆 | CC-20, CC-21 |
| 4 | **错误路径是一等公民** — 在有上下文的边界处理 | CC-153, PP-36 |
| 5 | **依赖向内流** — 业务逻辑不导入基础设施 | CA-12, CA-31 |
| 6 | **不过早抽象** — 三次法则：两次是巧合，三次是规律 | PP-15 |
| 7 | **童子军规则** — 碰到的文件顺手清理小脏东西 | CC-1 |

### 完成后检查清单

```
□ 构建通过
□ 测试通过（已有 + 新增）
□ 无新增警告
□ 无应该可配置的硬编码值
□ 无应该提取的重复逻辑
□ 提交信息描述清晰、范围明确
```

## 🔍 Reviewer 模式

当用户意图是评估代码时激活。

### 审查工作流

1. **定范围** — 确认审查对象：PR diff、模块、还是指定文件
2. **识别语言** — 确定范式；非 OOP 语言需调整规则适用性（见下方语言感知）
3. **逐项审查** — 按 10 点检查清单走查代码
4. **分级** — 为每个发现分配严重级别
5. **出报告** — 使用标准模板输出

### 10 点审查检查清单

#### 正确性与功能
- [ ] 逻辑正确实现了需求？(PP-75)
- [ ] 边界条件和错误处理完整？(CC-153, PP-36)
- [ ] 有安全漏洞？(PP-72)

#### 可读性与可维护性
- [ ] 命名揭示意图？(CC-4)
- [ ] 函数小且只做一件事？(CC-20, CC-21)
- [ ] 注释解释"为什么"而非"是什么"？(CC-39)

#### 设计与架构
- [ ] 遵循单一职责？(CA-8, CC-110)
- [ ] 避免重复 (DRY)？(PP-15, CC-37)
- [ ] 依赖方向正确？(CA-12, CA-31)

#### 测试
- [ ] 新代码有测试？(PP-91, CC-194)

### 语言感知

| 范式 | 语言 | Clean Code 适用度 |
|------|------|-------------------|
| 纯 OOP | Java, C# | ✅ 完全适用 |
| 多范式 | TypeScript, Python, Kotlin | ⚠️ 需调整 |
| 函数式 | Haskell, Elixir, F# | ⚠️ 多数规则不适用 |
| 系统/组合 | Rust, Go | ⚠️ 不同模式 |

### 严重级别

| 级别 | 标准 | 示例 |
|------|------|------|
| 🔴 **阻塞** | 安全漏洞、数据丢失风险、逻辑错误、生产路径崩溃 | SQL 注入、未验证鉴权、金额计算偏差 |
| 🟡 **警告** | 设计原则违反、指标超标、可维护性风险、关键路径缺测试 | SRP 违反、函数参数过多、核心逻辑无测试 |
| 🔵 **建议** | 风格偏好、小优化、可读性改善 | 命名可更精确、可提取常量 |

### 报告模板

```markdown
## 📋 代码审查报告

**审查范围:** [文件/提交]
**结论:** [✅ 可合并 / ⚠️ 需修改 / 🚫 需重做]

### 🔴 阻塞问题（必须修）
- **[文件:行号] 问题描述**
  - 规则: XX-## (规则名)
  - 原因: 为什么这是问题
  - 建议: 如何修复
  - 修复成本: [低/中/高]
  - 修复收益: [低/中/高]

### 🟡 警告问题（应该修）
- **[文件:行号] 问题描述**
  - 规则: XX-## (规则名)
  - 原因: 为什么这是问题
  - 建议: 如何修复
  - 修复成本: [低/中/高]
  - 修复收益: [低/中/高]

### 🔵 建议（可以修）
- **[文件:行号] 建议描述**
  - 建议: 改进方案
```

### 结论判定

| 结论 | 条件 |
|------|------|
| 🚫 需重做 | ≥3 个阻塞问题 或 根本性设计问题 |
| ⚠️ 需修改 | 任何阻塞问题 或 >2 个警告问题 |
| ✅ 可合并 | 零阻塞 且 ≤2 个警告问题 |

## 反模式清单（双角色共拒）

| 反模式 | 危害 | 正确做法 | 规则 |
|--------|------|----------|------|
| 复制粘贴微调 | N 个真相源 | 提取共享逻辑，参数化差异 | PP-15 |
| "能跑就行" | 忽视可维护性 | 能跑 + 清晰 + 有测试 | CC-1 |
| 治标不治本 | 技术债指数增长 | 失败两次后诊断根因 | PP-36 |
| 跳过文档/spec | 意图与实现渐行渐远 | 先读后写，永远 | PP-75 |
| 过早优化 | 无度量地增加复杂度 | 先 profile，再优化 | PP-63 |
| 上帝对象/函数 | 无法独立测试修改 | 单一职责 | CA-8 |
| 字符串类型接口 | 无编译期安全 | 枚举、类型化配置、schema | CC-175 |
| 无视现有模式 | 制造不一致 | 匹配代码库，改约定另开 PR | CC-1 |

## 语言无关质量门

任务"完成"前必须通过：

1. **编译/构建** — 零错误，零新增警告
2. **测试通过** — 已有 + 新增全绿
3. **无密钥入码** — API key、密码、token 只在 env/config 中
4. **无无主 TODO** — 每个 TODO 关联追踪项
5. **依赖锁版本** — 精确版本，不用范围
6. **文档同步** — 行为变了，文档跟着变
7. **Git 卫生** — 原子提交、描述性消息、功能分支

## 不审查的内容（机器的活）

以下应由 Linter/Formatter 处理：
- 格式和缩进
- 基础命名规范（大小写风格）
- 未使用的变量/导入
- 基础语法错误

**专注机器做不了的：** 逻辑正确性、设计决策、架构对齐。
