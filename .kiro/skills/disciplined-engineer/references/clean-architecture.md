## 第三部分：设计原则 (SOLID)

- **CA-8** SRP — 单一职责原则
  - 定义：一个模块应该只对一个行为者负责
  - 判断：这个模块有几个"变更理由"？如果不同角色（用户、运维、财务）会因不同原因要求修改同一模块，则违反
  - 违反信号：一个 Service 类同时处理业务逻辑、数据格式化、权限检查

- **CA-9** OCP — 开闭原则
  - 定义：对扩展开放，对修改关闭
  - 判断：添加新行为是否需要修改现有代码？
  - 实现手段：策略模式、插件架构、依赖注入
  - 违反信号：每次加新类型都要改 if/switch 链

- **CA-10** LSP — 里氏替换原则
  - 定义：子类型必须能替换其基类型而不破坏程序正确性
  - 判断：用子类替换父类后，所有调用方是否仍然正确工作？
  - 违反信号：子类抛出父类未声明的异常；子类的前置条件更严格

- **CA-11** ISP — 接口隔离原则
  - 定义：不应强迫客户端依赖它不使用的方法
  - 判断：实现类是否有空方法或 `throw NotImplemented`？
  - 违反信号：胖接口；实现类只用了接口 10 个方法中的 3 个

- **CA-12** DIP — 依赖倒置原则
  - 定义：高层模块不依赖低层模块，都依赖抽象
  - 判断：业务逻辑是否直接 import 了数据库驱动、HTTP 客户端、文件系统？
  - 违反信号：`UserService` 直接 `import psycopg2`；领域层引用框架类

## 第四部分：组件原则

- **CA-20** REP — 复用/发布等价原则
  - 复用的粒度等于发布的粒度。能独立发布的才能独立复用。

- **CA-21** CCP — 共同闭包原则
  - 一起变的类放在同一组件中。
  - 判断：一个需求变更是否只影响一个组件？如果要改多个组件，则组件划分有问题。

- **CA-22** CRP — 共同复用原则
  - 一起用的类放在同一组件中。不一起用的不要放一起。
  - 违反信号：引入一个包只为了用其中一个类，却被迫依赖整个包。

- **CA-23** ADP — 无环依赖原则
  - 组件依赖图中不能有环。
  - 判断：画出 import 关系图，是否存在 A→B→C→A？
  - 解决：依赖倒置（引入接口打破环）或合并组件。

- **CA-24** SDP — 稳定依赖原则
  - 依赖方向应朝向更稳定的组件。
  - 判断：被很多组件依赖的组件是否频繁变更？如果是，则架构不稳定。

- **CA-25** SAP — 稳定抽象原则
  - 越稳定的组件应越抽象。
  - 判断：核心稳定组件是否主要由接口/抽象类组成？

## 第五部分：架构

- **CA-31** 依赖规则
  - **核心规则：源码依赖只能指向内层。**
  - 层次（由外到内）：框架 → 接口适配器 → 用例 → 实体
  - 判断：内层代码是否 import 了外层的任何东西？
  - 违反信号：Entity 引用了 Flask/Express/Spring 的类

- **CA-32** 实体层
  - 企业级业务规则。最内层。不依赖任何外层。
  - 内容：领域对象、值对象、业务规则函数

- **CA-33** 用例层
  - 应用级业务规则。编排实体完成特定用例。
  - 内容：用例/服务类、输入/输出 DTO

- **CA-34** 接口适配器层
  - 转换数据格式。连接用例层和外部世界。
  - 内容：Controller、Presenter、Repository 实现、Gateway

- **CA-35** 框架与驱动层
  - 最外层。所有细节：数据库、Web 框架、UI、设备。
  - 原则：框架是工具，不是架构的核心。

## 关键洞察

- **CA-40** 框架是细节，不是架构
  - 判断：如果换掉框架，业务逻辑需要改多少？理想答案：零。

- **CA-41** 数据库是细节
  - 判断：业务逻辑是否直接依赖 SQL 语法或 ORM 特性？

- **CA-42** Web 是细节
  - 判断：业务逻辑是否依赖 HTTP 概念（request、response、session）？

- **CA-43** 好架构允许推迟决策
  - 判断：能否在不选定数据库/框架的情况下编写和测试业务逻辑？

- **CA-48** 测试是系统的一部分
  - 测试代码和生产代码同等重要。测试架构也需要设计。
