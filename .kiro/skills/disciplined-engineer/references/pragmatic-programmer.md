# The Pragmatic Programmer 规则参考 (Hunt & Thomas, 1999/2019)

按需加载。AI 在审查中引用 PP-## 时查阅此文件获取判断标准。

---

## 第 1 章：务实哲学

- **PP-1** 关心你的技艺
  - 如果不在乎做好，为什么还要花时间开发？

- **PP-2** 思考！你的工作
  - 判断：是否在自动驾驶模式下写代码？每个决策都应该是有意识的。

- **PP-3** 提供选择，不找借口
  - 当出问题时，提供解决方案而非解释为什么做不到。

- **PP-4** 不要容忍破窗户
  - 判断：代码中是否有已知的"坏味道"被放任？
  - 规则：看到坏代码就修，至少标记 TODO + ticket。放任一个破窗会导致整体腐化。

## 第 2 章：务实方法

- **PP-10** 正交性 — 消除无关事物之间的影响
  - 判断：改动模块 A 是否需要同时改动模块 B？
  - 测试方法："如果我改了这里，还有什么会受影响？"
  - 违反信号：一个 bug fix 需要改 5 个文件；UI 变更导致数据库迁移

- **PP-14** 可逆性 — 没有最终决策
  - 判断：当前设计是否把自己锁死在某个选择上？
  - 实践：用接口隔离第三方依赖；配置外置；避免供应商锁定
  - 违反信号：业务逻辑中硬编码了特定云服务的 API

- **PP-15** DRY — 不要重复自己
  - 定义：每条知识在系统中有且仅有一个权威表示
  - 四种重复类型：
    1. **强加的重复** — 环境要求（如多语言接口定义）→ 用代码生成消除
    2. **无意的重复** — 没意识到已经存在 → 改善发现机制
    3. **无耐心的重复** — 赶时间复制粘贴 → 纪律问题
    4. **开发者间的重复** — 团队成员不知道彼此的代码 → 改善沟通
  - 注意：**偶然相似不是重复**。两段代码碰巧长得像但代表不同业务知识，不应合并。

- **PP-17** 消除无关事物之间的影响（同 PP-10 的扩展）
  - 设计正交系统的好处：提高生产力（改动局部化）、降低风险（问题不扩散）

- **PP-27** 曳光弹
  - 用可工作的端到端代码探索未知，而非写完整规格再实现。
  - 适用场景：新技术、新架构、不确定需求
  - 与原型的区别：曳光弹是真实代码的骨架，原型是用完就扔的。

## 第 3 章：基本工具

- **PP-36** 死程序不说谎
  - 规则：崩溃优于继续运行在错误状态。
  - 判断：catch 块是否吞掉了异常然后继续执行？
  - 违反信号：空 catch 块；`catch (e) { /* ignore */ }`

- **PP-40** 资源使用完毕要释放
  - 判断：打开的文件/连接/锁是否在所有路径（包括异常路径）上都被释放？
  - 实践：try-with-resources、using、defer、context manager

## 第 4 章：务实偏执

- **PP-43** YAGNI — 你不会需要它
  - 判断：这个抽象/功能是否解决了当前实际存在的问题？
  - 违反信号："以后可能需要"的接口；没有第二个实现的抽象工厂
  - 平衡：YAGNI 不是不设计，而是不为假想的未来过度设计

- **PP-46** 迪米特法则 — 减少耦合
  - 规则：只和直接朋友说话。不要穿透对象图。
  - 判断：方法调用链是否超过 2 层？
  - 豁免：流式 API（Builder、Stream）是设计意图

- **PP-47** 避免全局数据
  - 判断：是否有模块级/全局变量被多处读写？
  - 违反信号：全局可变单例；模块级 `let` 被多个函数修改

- **PP-48** 如果全局数据重要到不能丢，就把它包装起来
  - 实践：用类/模块封装全局状态，提供受控的访问接口

- **PP-51** 继承层级不超过 2 层
  - 判断：继承链是否超过 Parent → Child → Grandchild？
  - 替代：组合、mixin、trait、接口

## 第 5 章：弯曲或折断

- **PP-57** 并发安全
  - 判断：共享可变状态是否有同步保护？
  - 违反信号：多线程访问同一集合无锁；async 函数修改外部变量

- **PP-63** 算法复杂度
  - 判断：是否了解关键路径的时间/空间复杂度？
  - 规则：对热路径的算法要能说出 O(?) 量级
  - 违反信号：循环内嵌套循环处理大数据集且无注释说明为何可接受

- **PP-64** 测试你的估算
  - 规则：性能假设要用实际数据验证，不要凭直觉

## 第 7 章：编码时

- **PP-72** 保持简单，最小化攻击面
  - 判断：是否暴露了不必要的接口/端点/权限？
  - 实践：最小权限原则；默认拒绝；输入验证

- **PP-73** 安全基本原则
  - 检查项：输入验证、参数化查询、最小权限、加密存储敏感数据
  - 安全问题无论项目级别都是 🔴 阻塞

- **PP-74** 命名好
  - 规则：名称是设计的一部分。好名字减少注释需求。
  - 判断：名字是否在 6 个月后仍然准确？

- **PP-75** 逻辑正确实现需求
  - 判断：代码行为是否与 spec/需求文档一致？
  - 这是最基本的正确性检查

## 第 8 章：项目前

- **PP-91** 测试你的软件，否则用户来测
  - 判断：新代码路径是否有对应测试？
  - 规则：没有测试的代码是未验证的假设

- **PP-92** 属性测试
  - 用不变量（invariant）验证代码，而非仅用具体用例
  - 适用：数据转换、序列化/反序列化、数学运算

- **PP-97** 取悦用户，不只是交付代码
  - 判断：这个实现是否真正解决了用户的问题，还是只满足了字面需求？

- **PP-100** 在你的作品上签名
  - 为你的代码负责。写出你愿意署名的代码。
