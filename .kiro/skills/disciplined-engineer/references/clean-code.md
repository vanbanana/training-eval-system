# Clean Code 规则参考 (Robert C. Martin, 2008)

按需加载。AI 在审查中引用 CC-## 时查阅此文件获取判断标准。

---

## 第 2 章：有意义的命名

- **CC-4** 名称应揭示意图
  - 判断：读名字能否不看实现就知道它做什么？
  - 违反信号：需要注释解释变量/函数用途
  - 示例：`d` → `elapsedTimeInDays`

- **CC-5** 避免误导
  - 判断：名称是否暗示了错误的类型或行为？
  - 违反信号：`accountList` 实际是 Set；`hp` 既像 hypotenuse 又像 health points

- **CC-6** 做有意义的区分
  - 判断：两个相似名称能否仅凭名字区分？
  - 违反信号：`ProductInfo` vs `ProductData`；`getAccount` vs `getAccountInfo`

- **CC-7** 使用可发音的名称
  - 判断：能否在口头讨论中自然说出？
  - 违反信号：`genymdhms`、`cstmrLst`

- **CC-8** 使用可搜索的名称
  - 判断：能否用编辑器搜索找到所有引用？
  - 违反信号：单字母变量（循环变量除外）、数字字面量

- **CC-14** 类名用名词，方法名用动词
  - 判断：类名是否描述"是什么"，方法名是否描述"做什么"？
  - 违反信号：类名 `ProcessManager`（动词+名词混合）；方法名 `data()`（名词）

## 第 3 章：函数

- **CC-20** 函数应该小
  - 判断标准：逻辑行数（排除空行、注释、文档字符串）
  - L3 阈值：≤50 行；L4：≤30 行；L5：≤20 行
  - 豁免：纯数据构建器、大型 switch/match、配置映射

- **CC-21** 函数只做一件事
  - 判断：能否用一句话描述函数做什么，且不用"并且"？
  - 违反信号：函数名含 `And`；函数内有明显的"阶段"分隔

- **CC-22** 每个函数一个抽象层级
  - 判断：函数内的操作是否在同一抽象层？
  - 违反信号：高层业务逻辑和低层字符串操作混在一起

- **CC-24** 避免 switch 语句
  - 适用：纯 OOP 语言（Java, C#）
  - 不适用：函数式语言的 match/when 表达式、TypeScript 的 discriminated union
  - 替代方案：多态、策略模式

- **CC-26** 函数参数越少越好
  - 理想：0 个（niladic）；可接受：1-2 个；需警惕：3 个；避免：>3 个
  - L3 阈值：≤5 个；L4：≤3 个
  - 豁免：大部分参数有默认值时只计必填参数；框架控制的 Builder/Factory

- **CC-37** 不要重复自己 (DRY)
  - 判断：同一知识是否出现在多处？
  - 注意区分：**真正重复**（一处改必须另一处也改）vs **偶然相似**（碰巧代码像但代表不同业务知识）
  - DRY 容忍度：L3 允许最多 3 次重复，第 4 次出现时报告

## 第 4 章：注释

- **CC-39** 好注释的类型
  - 法律注释、意图解释、警告后果、TODO（关联 ticket）、放大重要性

- **CC-43** 坏注释的类型
  - 多余注释（代码已经说清楚了）、误导性注释、日志式注释、归属注释（用 git blame）

- **CC-58** 注释掉的代码
  - 规则：删除它。版本控制会记住。
  - 判断：任何被注释掉的代码块都是违反

## 第 6 章：对象与数据结构

- **CC-80** 迪米特法则 (Law of Demeter)
  - 规则：方法只应调用——自身对象、参数对象、自己创建的对象、自己持有的字段
  - 违反信号：`a.getB().getC().doSomething()`

- **CC-81** 避免火车残骸
  - 判断：链式调用超过 2 层？
  - 豁免：流式 API（Builder、Stream、LINQ）是设计意图

## 第 7 章：错误处理

- **CC-92** 不要返回 null
  - 替代：空集合、Optional/Maybe、Null Object 模式
  - 违反信号：调用方需要 `if (result != null)` 检查

- **CC-93** 不要传递 null
  - 规则：参数不应接受 null 除非 API 明确设计为此

- **CC-153** 使用异常而非错误码
  - 判断：函数是否通过返回值传递错误状态？
  - 适用：OOP 语言；Go/Rust 的 error 返回值是惯用法，不算违反

## 第 10 章：类

- **CC-109** 类应该小
  - 衡量标准：职责数量，不是行数
  - 判断：能否用一句话描述类的职责，且不用"并且"、"或者"？

- **CC-110** 单一职责原则 (SRP)
  - 判断：这个类有几个变更理由？
  - 违反信号：类名含 `Manager`、`Processor`、`Handler` 且做了多件事

## 第 17 章：代码异味与启发

- **CC-130** KISS — 保持简单
- **CC-137** 并发安全 — 共享可变状态需要同步
- **CC-144** 注释掉的代码 — 删除
- **CC-147** 参数过多 — 超过阈值需重构
- **CC-162** 未使用的变量/导入 — 删除（交给 linter）
- **CC-164** Feature Envy — 方法过度使用其他类的数据，应该搬过去
- **CC-173** switch 语句 — OOP 中用多态替代
- **CC-175** 魔法数字 — 用命名常量替代所有非 0/1/-1 的字面量
- **CC-178** 深层嵌套 — 超过 3 层缩进需要提取或提前返回

---

## 第 9 章：单元测试

- **CC-194** 每个概念一个断言 — 测试聚焦单一行为
- **CC-102** 测试应可读 — 测试是文档，命名说明场景
- **CC-106** 测试应独立 — 测试之间无顺序依赖、无共享可变状态
